Risks & Mitigations
想定される懸念点と対応策

4AI（ChatGPT・Gemini・SAP Joule for Consultants・Copilot）によるレビューで洗い出した懸念点。

技術 / 高 WalkMe AI Action BarのAPIは2026年5月GA時点で仕様の詳細確認が必要 ▾
提供開始直後のため、Action BarからJouleを呼び出す公式APIの詳細仕様が事前に確認できない。実装開始後に仕様変更が起きるリスクがある。（ChatGPT・Gemini指摘）
対応策： Phase 0でIAS/IPS・WalkMeテナントの基盤整備を完了。提供開始後の最初の2週間をGo/No-Go判定に充て、仕様確認してから本実装へ。SAP担当者1名がリリースノートを毎週モニタリング。WalkMeが両タブ（Salesforce / S/4HANA Fiori）に常駐することを事前確認。
技術 / 高 Fiori四半期アップデートでWalkMeのセレクタが壊れるリスク ▾
SAP Fioriは四半期アップデートでDOM構造が変わることがある。行番号依存のセレクタを使うと毎回修正が必要になる。（ChatGPT指摘）
対応策： stable anchorベース設計を採用：①data-sap-ui属性 / aria-label優先、②nth-child系行番号固定指定は禁止（DOM再描画で壊れる）、③「追加直後のフォーカス行」「末尾行」を追跡するアプローチを採用。「stable IDで万全」は言い過ぎで、安定アンカーを取れる箇所を取り、取れない箇所は相対認識で逃がす設計が現実的。Cloud ALMと連動した四半期回帰テストを実施。
業務 / 中 AIの回答を信頼して受注することへの担当者抵抗 ▾
熟練営業ほど「慣れたやり方の方が速い」「AIが間違えたら誰の責任か」という抵抗が生まれやすい。（Copilot指摘）
対応策： JouleはAI参考情報として根拠（参照時刻・伝票ID）を必須表示し、最終判断は担当者が行う設計を徹底。与信ブロック・保存はSAP標準ワークフローが担保。Action BarのSnooze機能で担当者に主導権を残す。まず在庫確認・納期計算など低リスク業務から導入。
業務 / 中 Jouleのネイティブコンテキスト引き継ぎAPIは未公開 ▾
Jouleは現状、会話履歴を外部APIへ出力する公式手段が未公開。Joule UIはiframe内で動作しており、親ウィンドウからのコンテキスト取得は制約がある。「Joule→Smart Walk-Thruへの会話コンテキスト継承」をネイティブAPIで実現するのは現時点では困難。（技術AI評価・Gemini指摘）
対応策（AI出力継承方式・3段階フェールバック）： 「会話の継承」ではなく「AI出力の継承」として設計する。実装は3段階: ①最小リスク: Joule出力をDOM上から取得しWalkMe Dataに保存 ②中リスク: Joule応答を定型フォーマット（JSON/Key-Value）で返すスキルに寄せパースを安定化 ③低リスク（前提あり）: client bridge APIでスレッド/応答を取得（GA直後は「at risk」領域）。いずれの方式でもフェールバックを必須とし、AIなしでもSWT/SmartTips中心で成立する導線を常に残す。
技術 / 高 WalkMeロード前にJouleアイコンをクリックすると別webclientが開く ▾
WalkMeがロードされる前にユーザーがJouleアイコンをクリックすると、WalkMe管理外の別webclientでJouleが開いてしまう。AI Action Barが「入口」として機能しなくなるリスク。（Joule-WalkMe Steering資料指摘）
対応策： ①WalkMeのロード完了までJouleアイコンの表示/クリックを制御 ②ロード順序をPhase 0で検証し、WalkMe常駐が確認できてから本実装へ ③万が一別webclientが開いた場合のユーザー向けガイド（「Action Barからご利用ください」）を用意
運用 / 中 WalkMe AI機能のOpt-in/Opt-out運用とAI Units消費 ▾
WalkMe FY26 AI Commercializationにより、AI機能はOpt-in/Opt-out運用。Joule統合時にAI Units消費が発生し、本番で急にAIが無効化される/有効化が遅れるリスクがある。（WalkMe FY26商用条件指摘）
対応策： ①AIなしでも成立する導線（SWT/SmartTips中心）を常に残す設計 ②AI Units消費量を事前に見積もり、予算承認を取得 ③Opt-in手続きのタイムラインをWalkMeベンダーと事前確認
SAP / 中 UKMの公開APIはSOAPが中心。OData可用性は要確認 ▾
SAP Credit ManagementのAPIは主にSOAPサービス（Credit Management Account Read、DCD Create/Update等）で提供。ODataとしての汎用読み取りエンドポイントは公開情報では確認できていない。（SAP Joule for Consultants評価）
対応策： BTP側でSOAPクライアントまたは中間サービスとして消費し、他API（ATP等）と集約する設計。クレジット統合用Communication Scenario（SAP_COM_0793/0826/0092）の活用を検討。Phase 0でAPI可用性を確認しGo/No-Go判定に含める。
変革管理 / 中 管理職（課長・部長）の巻き込みとKPI設計 ▾
現場担当者向けの設計は成熟しているが、管理職が気にする「説明責任」「監査耐性」「属人性リスク」への訴求が不足している。管理職の支持がなければ現場導入が進まない。（変革管理AI評価指摘）
対応策： ①管理職用ダッシュボードを設計（差し戻し理由Top3・例外案件の傾向・与信判断リードタイム） ②訴求メッセージは「部下を守る材料が増える」（監査耐性・説明責任の強化） ③KPIを4レイヤーで整理: 業務（差し戻し率・リードタイム）/ 品質（例外案件比率）/ 人（新人独り立ち期間）/ 組織（電話削減→付加価値業務転換時間）
SAP制約 / 低 IAS/IPSの信頼ドメイン設定不備で認証ループが発生 ▾
IAS・WalkMeの信頼ドメイン設定に抜け漏れがあると、Joule起動時に認証ループや空白画面が発生する。特にSalesforceとS/4HANAのクロスシステム遷移時に発生しやすい。（SAP Joule for Consultants指摘）
対応策： IAS Trust → BTP Trusted Domain → IAS Trusted Domain → CSP → Salesforce Trusted Domain の全5レイヤーを順番に設定。SAML/OIDC混在禁止（同一IASテナント上で混在不可）。SAML Tracerで認証フローを必ず検証。Phase 0のWeek 1でGo/No-Go判定し、IAS設定不備はNGとして本実装に入らない。追加注意: ①ブラウザのポップアップ抑止（ユーザー操作なしのwindow.open制限）への対応 ②SameSite Cookie/サードパーティCookie規制によるSSO不安定化 ③連続遷移時のセッション維持検証
