入力検証ゲート

SAP S/4HANA Cloud 伝票入力における部門コード × 勘定科目の不正組み合わせを、Jouleカスタムスキルの検証を通過して初めて保存可能にする「ゲート型」の設計。BAdIが使えない前提での補助的ゲートとして、Joule Action Bar + WalkMe Launcherの連携で実現する。

Scenario
佐藤さんの業務 — 伝票入力の「あるある」
ペルソナ佐藤さん（入社3年目・経理担当）
対象アプリManage Journal Entries (F0718)
業務製造部門の経費伝票入力

😤 導入前の痛み： 部門コードを入力後、共有フォルダのExcel対照表（v3_最終_山本修正.xlsx? v4_確定版(2).xlsx?）を探して科目を確認。見つからないと隣の先輩に「1000番台って何使えましたっけ？」と聞く。先輩も忙しいので「たぶん500000番台」と曖昧な回答。それを信じて保存→転記後に「製造部門に販売費」と発覚→修正仕訳→上長説明→再承認。1件の確認に2〜3分、差し戻し修正に15分。
✨ WowMoment： 画面を開いた瞬間にAction Barが「部門・科目の組み合わせをチェックできます」と提案。部門コードを入力→勘定科目を入力→「チェック」ボタン1クリック→1秒でOK/NG判定。NGなら正しい科目一覧がカードで表示。OKならLauncherが消えて保存ボタンが現れる。Excel対照表も先輩への質問も差し戻しも不要に。
Design Principle
「ブロック」ではなく「ゲート」

保存ボタンは最初からWalkMe Launcherで覆われた状態（保存不可）がデフォルト。Jouleカスタムスキルによる検証を通過して初めてLauncherが消え、保存可能になる。「間違えたら止める」のではなく「正しいことを確認してから通す」設計。

Key Verification Points
最重要検証ポイントと既知リスク

技術検証依頼書 + 4AI評価（SAP業務妥当性・技術実現性・変革管理・競合比較）で洗い出した懸念点。

最優先 Joule Action Bar内のボタンアクションからWalkMeAPI.setVariable()を呼び出せるか ▾
Joule Action BarはWalkMeのランタイム上で動作するため、同一JavaScriptコンテキスト内でWalkMe APIを呼び出すことは原理的に可能なはず。しかしGA直後のため、ボタンアクション設定でカスタムJS実行が許可されているかの実機確認が必要。
対応策： GA直後の最初の2週間でこの1点のみを集中検証。不可の場合はパターンB（OnScreen RulesによるDOM監視）にフォールバック。
高 Jouleカードの表示領域制限・スクロール可否 ▾
NG時に「利用可能な勘定科目一覧」を表示するのがこのシナリオの核心的な体験。カード内のコンテンツにmax-height制限があり一覧が切れる場合、業務的に使い物にならない。
対応策： 代替A: テキストで科目範囲を案内。代替B: WalkMe ShoutOut（従来型ポップアップ）で一覧表示。代替C: Jouleチャットに「使える科目を教えて」と聞く導線。
高 CDS View → OData → Jouleスキルのレスポンスタイム ▾
チェックボタンクリックから結果表示までの目標は2秒以内。CDS Viewの複雑さ、ODataの認証オーバーヘッド、Jouleスキル処理のそれぞれでボトルネックが発生しうる。
対応策： CDS Viewはマスタテーブル参照のみ（CEPC+SKA1で軽量）。$select最小化・$expand回避。Communication Arrangementの認証キャッシュ活用。
中 UI層ゲートの限界 — 開発者ツールでのバイパス・他チャネルからの投稿 ▾
4AI評価（SAP業務妥当性）で指摘：技術的に詳しいユーザーがDevToolsでLauncherを削除する可能性。また、テンプレートアップロード・API経由の仕訳投稿はUI層ゲートでは防げない。SAP標準のValidation/BAdIはサーバーサイドで全チャネルをブロック可能。
対応策： 本シナリオはBAdI対象外の制約下での補助的ゲートとして位置付け。実務上の一般ユーザーがDevTools操作を行うリスクは極めて低い。将来的にBAdI追加で完全対応可能。他チャネルの制御が必要な場合は、拡張パスとしてValidation（F4406/F2718）またはBAdI（FIN_CODING_BLOCK_VALIDATION）の追加を推奨。
中 導入直後の「邪魔されている」感とベテランの抵抗 ▾
4AI評価（変革管理）で指摘：Launcherは導入直後は「邪魔」と感じられる（Jカーブ効果）。ベテラン経理は「不要な補助輪」と映る可能性。月末繁忙期のワンステップ追加は抵抗を生みうる。
対応策： 呼称を「不正防止ゲート」→「仕訳アシスタント」に変更。ベテランには「若手からの質問攻めから解放する仕組み」として訴求。導入は3〜5名パイロット（2週間）→全体展開。エラー履歴は「個人評価には使用しない」と部門長から宣言。
